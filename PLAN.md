# Gitea-Jenkins Bridge (gitea-plugin-rs) Implementation Plan

## Назначение
Этот документ является основным планом реализации для проекта `gitea-plugin-rs`. Он предназначен для передачи контекста новым ИИ-сессиям (Antigravity/Claude), чтобы агенты сразу понимали, что уже сделано, а что предстоит.

## Текущий статус (Этап 1: Завершен)
Базовый скелет микросервиса, маршрутизация и маппинг событий полностью реализованы. Код оригинального Java-плагина (`jenkinsci/gitea-plugin`) изучен на 100%.

- `[x]` **Инициализация Cargo Workspace** (крейты: `gitea-client`, `jenkins-client`, `webhook-server`, `bridge-logic`)
- `[x]` **Крейт `gitea-client`**: Описаны модели (Repository, User, PushEvent, PullRequestEvent) и базовый REST-клиент.
- `[x]` **Крейт `jenkins-client`**: REST-клиент для Jenkins (получение CSRF crumb, триггер джобов).
- `[x]` **Крейт `bridge-logic`**: Реализован маппер (преобразование Gitea-вебхуков в триггеры Jenkins).
- `[x]` **Крейт `webhook-server`**: Axum-сервер, роутинг, валидация HMAC-подписи (`x-gitea-signature`), эндпоинты для вебхуков и коллбеков статуса.
- `[x]` **Аудит легаси кода**: Полностью проверены классы API, SCM Навигаторы, Webhook Handlers, GiteaAssetPublisher и UI-компоненты.

## Следующие шаги (Этап 2: Тестирование, QA и Деплой)

- `[x]` **Юнит-тесты (TDD)**: Покрыть тестами мапперы в `bridge-logic`. Тесты должны проверять различные сценарии ветвления и пулл-реквестов без сетевых вызовов.
- `[x]` **Интеграционное тестирование**: Настроить мок-серверы (например, с использованием `wiremock`) для тестирования `jenkins-client` и `gitea-client`.
- `[x]` **Логирование и метрики**: Улучшить использование крейта `tracing` во всех модулях (добавить span'ы для каждого входящего запроса).
- `[x]` **Докеризация**: Написать `Dockerfile` (multi-stage) и `docker-compose.yml` для тестирования Gitea, Jenkins и вебхук-сервера вместе.
- `[x]` **Интеграция E2E**: Проверить связку Jenkins + Gitea + bridge в Docker. Устранить несоответствия маппинга.

## Архитектурные решения и обработка Edge Cases
- **Конфигурация**: Отказ от глобальных настроек внутри Jenkins (`GiteaServers`). Вся конфигурация Rust-моста производится исключительно через `.env` переменные (`GITEA_TOKEN`, `JENKINS_URL` и др.).
- **Релизные ассеты (`GiteaAssetPublisher`)**: В микросервисной архитектуре мост не должен забирать артефакты из воркспейса Jenkins. Публикация релизов теперь должна осуществляться напрямую из пайплайна (`Jenkinsfile`) с помощью утилит типа `curl` или HTTP Request Plugin.
- **Валидация вебхуков**: Реализована строго через middleware на уровне Axum, отклоняя любые запросы без валидной HMAC-sha256 подписи.
